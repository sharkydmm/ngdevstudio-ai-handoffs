## ADR Summary
**1. Deal-submission limit model (`0`=blocked, `1000`=unlimited).** Resolution order at submission time: **user override** (if the user has a `DealLimits` row) **OR** **[ max(business `DealLimits`, active-package limit) — else country default ]**; blocked when `posted >= max`. `0` deliberately disables submission (e.g. US/SE non-deal markets); "unlimited" is the sentinel `1000`, never `0`. Documents the **L1 regression** (2025 BF reset never reverted for DE/GB/FR/PL/NL), what-must-not-break, and test coverage (`StoreTest::testStoreWithLimitReached` proves `0`=blocked; happy-path is test-blind/skipped). Refs: `ThreadService.php:1715,1742,1761-1769`, `BusinessService.php:646,665,668,753,760-763`, `config/countries.php`.

**2. Promotion eligibility (verified billing OR approved network).** For a **business contact**, promotions are allowed **per country** where the business has — via polymorphic `billingSettings` — **either** a **verified billing account** (`is_verified = VERIFIED`) **or** an **approved network** (`status = APPROVED`); internal users use the enabled-promotion-country set instead. It's **OR, not AND**, per country. Documents failure modes (PENDING/DENIED networks & unverified billing don't count; internal-vs-contact divergence; the rule is mirrored in `AdManager/BaseRequest` + `OrderBusinessCountryRule` and must stay in sync). Test coverage: enable/disable is tested, but **the eligibility OR itself is NOT-TESTED** (flagged; regression test recommended). Refs: `PromotionRestrictionSettings.php:20-45` (conditions at `:36`,`:38`), `VerificationStatus`(VERIFIED=1), `NetworkStatus`(APPROVED=2).
