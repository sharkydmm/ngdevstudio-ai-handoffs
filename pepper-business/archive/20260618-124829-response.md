## Recommendations

**Top 3 highest-value documentation improvements**
1. **Confidence & Risk Register** — *done*; keep it updated as the canonical findings index.
2. **`CODEOWNERS` + git-derived ownership map** — directly mitigates the bus-factor SPOF; trivial to produce from `git shortlog`.
3. **ADR: deal-submission limit model + BF-revert checklist** — converts the L1 root cause (a forgettable manual revert) into a durable, repeatable guard.

**Top 3 onboarding improvements**
1. Link the Risk Register into onboarding `README` / `first-week` so new engineers learn the known traps early.
2. A **service → external-system dependency map** (Metabase, GAM, Pepper B2C, networks, accounting) — fastest way to grasp blast radius.
3. Glossary entries for the limit model (`0`=blocked vs `1000`=unlimited) and "intentional-zero markets" (US/SE).

**Top 3 operational documentation improvements**
1. **BF-limit-change runbook** (set + dated revert checklist) — prevents L1 recurrence.
2. **Secrets rotation runbook** (GAM key incl. `private_key_id`, prod-DB cred, `.env.*`) — ready when access exists.
3. **Metabase card-contract register + a "money card changed?" check** in the production-support runbooks.

**What I would personally document next:** **`CODEOWNERS` + the ownership map.** It's fully repo-doable (`git shortlog -sn` per path), directly addresses a confirmed SPOF (B1), and unblocks every future "who do I ask / who reviews this" question — highest maintainability ROI with no external dependency.

**What documentation work now has low ROI and should stop:** net-new **narrative/explanatory documents** (more domain prose, more onboarding sub-guides, atlas-style expansions). The corpus is mature and code-strong; further prose has diminishing returns. Also stop re-deriving the paused findings (M1/L1/GAM) — they're now preserved; only *validation* (external) adds value.
