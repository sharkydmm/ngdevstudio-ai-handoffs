# Revenue-Critical Configuration Audit — Pepper Business

Branch: vex-na-update-docs · **Investigation only — no configuration changed, no MRs.**

## 1. Audit summary

Swept `config/*` (60+ files), with deep focus on the revenue-critical sources: `countries.php` (per-country limits/thresholds/accounting/currency), `appConfig.php` (global flags), `adManager.php`, `promotable_deals_thresholds.php`, invoice settings. Cross-checked findings against code usage and the docs corpus.

**Headline:** the DE/GB/FR/PL/NL `0` deal-limit issue is **not isolated** — the config layer has several latent risks: one **confirmed dead config**, the **0-limit regression**, a **fail-open mock default**, a **disabled moderation check**, and **hardcoded country logic in revenue code**. None are catastrophic in isolation; together they show config drift in the revenue/moderation path that deserves a dedicated cleanup pass.

Structural integrity is good: all 10 countries have all required keys (`invoice_threshold`, `max_deals_*`, `categories`, `accounting_app`, `currency_code` — 10/10 each). `promotable_deals_thresholds` (production) is uniform and healthy (no `0`s).
