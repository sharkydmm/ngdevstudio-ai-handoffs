# L1 Impact Quantification — Measurement Plan & Production Queries

Branch: vex-90-vue-migration-changes (docs/handoff on vex-na-update-docs) · **Investigation/planning only — no changes.** Repo investigation is complete; this is the shortest path to "how much real business activity was affected."

## Critical measurement constraint (read first)
**Blocked submission attempts leave NO database row.** `ThreadService::store()` throws `ThreadLimitReachedException` → HTTP 406 **before** any `threads` insert (verified: the exception is not logged to any table). Therefore:
- The DB can measure the **drop in *successful* submissions** (before vs after) and the **population at risk** — strong, sufficient signals.
- The DB **cannot** count blocked attempts directly. The only source of "attempted-but-blocked" counts is **Bugsnag/Atatus (406s / exception telemetry)** or web access logs — out of DB scope.

## Production Queries (read-only, copy/paste ready)

Run on a read replica if available. All are pure `SELECT` / `GROUP BY` — no writes. `thread_type=1` = DEAL.

```sql
-- V2: Monthly successful deal submissions by country (affected vs control). Look for a drop after 2025-10.
SELECT country,
       DATE_FORMAT(created_at, '%Y-%m') AS ym,
       COUNT(*) AS deals
FROM threads
WHERE thread_type = 1
  AND created_at >= '2025-06-01'
  AND country IN ('DE','GB','FR','PL','NL','AT','ES','MX')
GROUP BY country, ym
ORDER BY country, ym;
```

```sql
-- V3: Pre vs post window totals + drop ratio per country (regression went live ~2025-10-31).
SELECT country,
       SUM(created_at >= '2025-08-01' AND created_at < '2025-10-31') AS deals_pre,   -- 3 months before
       SUM(created_at >= '2025-10-31')                              AS deals_post,   -- since the commit
       ROUND(SUM(created_at >= '2025-08-01' AND created_at < '2025-10-31') / 3, 1)   AS avg_monthly_pre
FROM threads
WHERE thread_type = 1
  AND created_at >= '2025-08-01'
  AND country IN ('DE','GB','FR','PL','NL','AT','ES','MX')
GROUP BY country
ORDER BY country;
```

```sql
-- V4: POPULATION AT RISK — active businesses enabled to submit deals in the 5 affected
-- countries that have NO effective non-zero business-level limit (so they fall back to the 0 default).
-- (Package-granted limits are a secondary exemption — see V5.)
SELECT e.country, COUNT(*) AS businesses_at_risk
FROM business_deal_submission_enabled_countries e
JOIN businesses b
  ON b.id = e.business_id AND b.status = 1          -- ACTIVE only
LEFT JOIN deal_limits dl
  ON dl.affected_type LIKE '%Business'
 AND dl.affected_id = e.business_id
 AND dl.country = e.country
 AND dl.max_deals_per_week > 0
 AND dl.start_date <= NOW()
 AND (dl.end_date IS NULL OR dl.end_date >= NOW())
WHERE e.country IN ('DE','GB','FR','PL','NL')
  AND dl.id IS NULL                                  -- no active non-zero business override
GROUP BY e.country
ORDER BY e.country;
```

```sql
-- V5: OVERRIDE PREVALENCE — active businesses enabled in the 5 countries that DO have a
-- non-zero, currently-effective business-level limit (exempt from the 0 default).
SELECT e.country,
       COUNT(DISTINCT e.business_id) AS enabled_businesses,
       COUNT(DISTINCT CASE WHEN dl.id IS NOT NULL THEN e.business_id END) AS with_business_override
FROM business_deal_submission_enabled_countries e
JOIN businesses b ON b.id = e.business_id AND b.status = 1
LEFT JOIN deal_limits dl
  ON dl.affected_type LIKE '%Business'
 AND dl.affected_id = e.business_id
 AND dl.country = e.country
 AND dl.max_deals_per_week > 0
 AND dl.start_date <= NOW()
 AND (dl.end_date IS NULL OR dl.end_date >= NOW())
WHERE e.country IN ('DE','GB','FR','PL','NL')
GROUP BY e.country
ORDER BY e.country;
```

```sql
-- V5b: Secondary exemption — any active package-level deal limits for these countries
-- (these would also bypass the 0 default for businesses holding such a package).
SELECT country, COUNT(*) AS active_package_limits
FROM deal_limits
WHERE affected_type LIKE '%Package'
  AND country IN ('DE','GB','FR','PL','NL')
  AND max_deals_per_week > 0
  AND start_date <= NOW()
  AND (end_date IS NULL OR end_date >= NOW())
GROUP BY country;
```

```sql
-- V6: Distinct submitting businesses, pre vs post — did real submitters disappear?
SELECT country,
       COUNT(DISTINCT CASE WHEN created_at >= '2025-08-01' AND created_at < '2025-10-31'
                           THEN business_id END) AS distinct_submitters_pre,
       COUNT(DISTINCT CASE WHEN created_at >= '2025-10-31'
                           THEN business_id END) AS distinct_submitters_post
FROM threads
WHERE thread_type = 1
  AND created_at >= '2025-08-01'
  AND country IN ('DE','GB','FR','PL','NL','AT','ES','MX')
GROUP BY country
ORDER BY country;
```

If the `threads` table is huge, add an index hint or run on a replica.

## Risks
- **Interpretation risk:** a post-Oct drop could partly be seasonal — **the AT/ES/MX controls are essential** to attribute it to the regression.
- **Under-counting impact:** successful-submission drop **understates** harm (blocked attempts are invisible in DB) — V4 + Bugsnag 406s capture the rest.
- **Deploy-date skew:** use the real prod deploy date of `842d7ec58`, not just the commit date.
