## What changed & why
The TradeDoubler validation client used to authenticate with a login-form POST
and a scraped `PHPSESSID` session cookie against the internal publisher portal
(`/en/publisher/api/programs`). The portal session TTL is only a few hours, so
automated daily validation kept breaking. TradeDoubler support confirmed an
official Publisher API exists for this use case (VEX-88):

```
GET https://api.tradedoubler.com/publisher/programs?sourceId={id}&statusId=3&limit={n}
Authorization: Bearer {token}
→ { "items": [ { "id", "name", "statusId", ... } ] }
```

`TradeDoublerApiClient` was rewritten to use it, bringing it in line with the
Awin/CJ bearer-token pattern.

### Before → After
- **Auth:** login POST + session cookie (CachesApiToken trait, CookieJar, 302
  handling, stale-session/HTML detection) → **static bearer token** via
  `withToken()`. `requiresTokenCaching()` is now `false` (like Awin/CJ).
- **Endpoint:** internal portal host → official `https://api.tradedoubler.com/publisher/programs`.
- **Source scoping:** the official endpoint scopes by a single `sourceId`, so the
  client calls it once per comma-separated source id and **merges/dedupes** the
  results — one credential row still covers all of a country's sources.
- **Errors:** real upstream HTTP status is preserved so the validation service's
  retry rule (429/5xx → retry; 401/4xx → INVALID) keeps working.
- **Cache:** the per-batch programs-list cache is kept purely as a fetch-sharing
  optimisation (not a token cache).
- **Resolver semantics unchanged:** in accepted list (`statusId=3`) → VALID,
  absent → INVALID.

`CachesApiToken` / `FreshTokenResult` are untouched (still used by Adcell).
