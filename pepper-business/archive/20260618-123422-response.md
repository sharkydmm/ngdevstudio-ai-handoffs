# Staff-Engineer Confidence Audit — NEW finding: Billing/reporting integrity depends on BI-editable Metabase column *names*

Branch: vex-90-vue-migration-changes (docs/handoff on vex-na-update-docs) · **Investigation only — no changes.** GAM and L1 remain **paused** (no new evidence); this is a fresh, repo-discovered unknown.

## Headline finding — M1 (NEW)

**The entire invoice + payment + report pipeline silently couples to Metabase card *column display-names*, which are owned and editable by the BI team outside this repository — with no validation, no caching, no alerting, and tests that only mock Metabase.** A casual BI-side column rename or query edit can silently corrupt invoice amounts (or break generation) with **zero signal to engineering**.

### Why it matters
Metabase is the **source of truth for all money** (invoices, payments, loyalty payouts, CPM/CPC reports). The coupling to it is the most fragile kind possible — by human-readable **column display name** — and crosses a **team boundary** (BI edits the cards; engineering hardcodes the IDs and the expected column names). This is a classic "production behaviour depends on configuration rather than code," "tests exist but don't prove the business behaviour," and "revenue assumption never measured" — all at once.

### Evidence (code-proven)
- **Source of truth = 14 hardcoded card IDs:** `config/metabase.php:7-21` — `cpc=7220, cpm=7097, invoices=7222, cpmInvoices=7291, payments=7261, loyaltyProgramInvoices=8442`, + report cards. Not env-driven; card *queries* live in Metabase.
- **Coupling is on the card's `display_name`:** `MetabaseApiClient::mapReportColumnKeys()` →
  ```php
  if (isset($this->columnMap[$column['display_name']])) { ... } else { $mappedKeys[] = $column['display_name']; }
  $item = array_combine($columns, $row);   // keys = mapped display names
  ```
  The billing contract is literally the card's column titles. Example (`CpmReports::$columnMap`):
  ```php
  'revenue' => 'commission',           // money column → billing field
  'paid_unique_clicks' => 'paid_clickouts',
  'impressions' => 'impressions', 'paid_thread_visits' => 'paid_deal_visits', ...
  ```
  Rename the card's `revenue` column → fallthrough to raw name → the `commission` key is never produced → billing reads null → **silent under-billing**.
- **No schema validation:** `getMetabaseData()` throws only a generic `Exception` if `data` is absent; `extractReportData()` does `array_combine` with **no check that required money columns exist**.
- **Response caching is disabled** (entire `Cache::remember` block commented in `MetabaseApiClient::getData()`), so invoice generation has a **hard live dependency on Metabase availability at cron time** — no fallback. (Auth token is cached; payloads are not.)
- **No alerting/monitoring** on Metabase results (none in `Kernel`/`Notifications`).
- **Tests mock Metabase** (`MetabaseMockData`, prior pass) → they validate the app's math against *fabricated* columns, never the live card contract.

## Risks
- **Silent revenue corruption (High likelihood over time):** BI edits cards routinely; nothing prevents a rename/query change from skewing invoices. The failure is *plausible, silent, and cross-team*.
- **Invoice-run fragility (Medium):** no cached fallback → Metabase availability at the Nth-business-day cron is a single point of failure.
- **False confidence from tests:** green finance tests do **not** prove the live card contract.
