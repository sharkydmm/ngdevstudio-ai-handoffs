# Decision Report — Intended meaning of a `0` deal-submission limit (Pepper Business)

Branch: vex-na-update-docs · Investigation only — no code changed, no tests re-enabled.

## 1. Investigation summary

Evidence gathered from git history, commit messages, ticket IDs, the comparison-logic evolution, the migration, tests, and ADRs. The picture is **consistent and strong**: a configured limit of **`0` means "zero deals allowed" (blocked)** — it is a deliberate configuration for non-deal markets, **not** a sentinel for "unlimited". "Unlimited" is expressed by a **large number (1000)**.

Decisive evidence:
- **Explicit business decisions in commit messages:**
  - `fbfdc1575` (2025-11-07) *"Adding United States (No GAM, **0 deal submission**, main usage will be Clicky)"* → sets `US` to `0/0`.
  - `f3e4b361f` (2024-11-21) *"Adding Sweden (No GAM, **0 deal submission**, main usage will be event promotions)"* → sets `SE` to `0/0`.
- **Implementation has never treated 0 as unlimited:** `ThreadService::userLimits` and `BusinessService::checkIfLimitReached` both return `posted >= max`. With `max=0`, `0 >= 0` is true → blocked. `PP-274` ("Wrong deal submission logic", `04dd4f3fa`) refined *which* limits feed the check (`max(package, business)` else country default) but kept the `>=` semantics — no `>0` guard ever existed.
- **"Unlimited" convention = 1000:** the BF commit changed DE/GB from `1000 → 0`; `1000` is the de-facto "no practical cap" value for high-volume deal markets.
- **Test corroboration:** `Thread/StoreTest::testStoreWithLimitReached` sets a `0/0` business limit *specifically to force* `ThreadLimitReachedException` — i.e., the test author treats `0` as "blocked".
- **No "unlimited"/"no limit" semantics anywhere** in `app/`, `config/`, `database/` (only unrelated hits). Migration `2019_08_15` declares `max_deals_per_week` as a plain `integer`, no default, no comment.

## 3. Confidence assessment

| Question | Answer | Confidence | Evidence class |
|---|---|---|---|
| What does `0` mean? | **Blocked — zero deals allowed** | **High** | Explicit business decision (US/SE commits) + implementation + test expectation |

| Are DE/GB/FR/PL/NL `0` intended? | **Probably NOT — incomplete revert** | **Medium-High** for DE/GB (were `1000`); **Medium** for FR/PL/NL | Pre-BF non-zero + revert intent "back to defaults" + `0`-convention is for non-deal markets only |
| Are SE/US `0` intended? | **Yes** | **High** | Explicit commit messages |

**Contradicting evidence:** none found that `0` = unlimited. The only tension is *config correctness* (DE/GB/FR/PL/NL still `0`), not the *meaning* of `0`.
**Unknown:** the Jira tickets (PP-274/PP-2678/PP-2702/PP-NA) were read via commit messages only — the Jira bodies themselves were not accessed; and whether DE/GB/FR/PL/NL `0` is truly unintended needs product confirmation + a prod check of live submissions.

## 4. Recommended decision (ADR-style)

**Decision:** Treat **`0` = "deal submission disabled" (blocked)** as the authoritative, intended semantics across thread limits, user limits, and country configuration. Encode this as an ADR. **"Unlimited" must be expressed as a large value (e.g., `1000`), never `0`.**

**Corollary (config-correctness, separate from semantics):** `DE, GB, FR, PL, NL = 0` is **most likely an incomplete BF revert** and a **live revenue risk** (businesses without an explicit business/package limit cannot submit deals there). This requires product confirmation before any change — **do not edit config blind**.

**Consequence for the skipped Cluster A tests:** they should **not** rely on the country default. The correct, production-safe repair is to **grant an explicit non-zero business limit in the fixture** (`DataManager::addBusinessLimit(...)`) before creating deals — decoupling the tests from the ambiguous config entirely.
